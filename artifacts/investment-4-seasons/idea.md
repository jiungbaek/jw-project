# AI 투자 4계절 웨더캐스터

## Problem Statement
HMW turn a single LLM prompt into the *entire* backend for a real-time macro signal — so that "오늘의 투자 계절"을 판정하는 데 스크래퍼도, DB도, cron도 필요 없게 만들 수 있을까?

## Recommended Direction
"실시간 매크로 계절 브리핑" — 캐주얼한 데일리 콘텐츠 톤으로, Gemini API의 실시간 검색(grounding)이 오늘 기준 CPI·미국채 금리·주요 지수 추세를 읽고 이를 봄/여름/가을/겨울 4계절 중 하나로 판정한다. 크롤러 대신 LLM의 실시간 검색을 유일한 데이터 소스로 써서, 3시간 안에 스크래퍼 구현·유지보수 없이 완결되는 구조를 만든다.

정확한 수치를 보증하는 도구가 아니라 "그럴듯한 방향성 판정 + 위트있는 카피"를 제공하는 재미 중심 콘텐츠로 포지셔닝한다. 이 프레이밍이 LLM 실시간 검색의 수치 정확성 리스크를 프로덕트 결정으로 흡수한다. 메인 페이지(소개+CTA)와 결과 페이지(계절 판정+근거) 2개 화면으로 구성하고, Vercel에 배포해 데모 가능한 상태를 3시간 내 완주 기준으로 삼는다.

## Key Assumptions to Validate
- [ ] Gemini API가 실시간 검색(grounding)으로 오늘 날짜 기준 CPI·미국채 금리·주요 지수 추세를 구조화된 형태로 반환한다 — UI 작업 전 스크립트로 API 호출 먼저 찍어서 검증
- [ ] 응답 포맷(계절 판정 + 근거)이 매 호출마다 파싱 가능한 수준으로 일관된다 — 동일 프롬프트로 3-5회 반복 호출해 포맷 흔들림 확인
- [ ] 방향성 위주의 캐주얼 판정이어도 사용자가 결과를 그럴듯하다고 받아들인다 — 별도 검증 없이 데모 반응으로 판단(3시간 스코프 밖)

## MVP Scope
- 메인 페이지: 서비스 소개 + "오늘의 계절 확인하기" CTA
- 결과 페이지: Gemini 호출 → 봄/여름/가을/겨울 중 판정 + 판정 근거(지표별 한 줄 요약)
- 요청마다 Gemini 실시간 검색을 새로 호출 (캐싱 없음, 결과가 매번 조금 달라도 허용)
- 로그인/이력/DB 없는 완전한 stateless 구조
- Vercel 배포까지 완료된 상태가 완주 기준

## Not Doing (and Why)
- 크롤링/자체 데이터 파이프라인 — 3시간 내 구현·유지보수 비용이 가장 큰 낭비 지점이라 의도적으로 회피
- 로그인, 사용자별 이력 저장, DB — 3시간 stateless 데모에 불필요한 복잡도
- 결과 캐싱 — 같은 날 재요청 시 판정이 달라져도 허용하기로 결정했으므로 캐싱 인프라가 불필요
- CPI/금리/지수 외 추가 지표(고용지표, PMI 등) — 스코프를 3개 지표로 고정해 프롬프트와 UI를 단순하게 유지
- SNS 공유 카드 이미지, 계절별 자산배분/투자전략 추천 — 재미 중심 톤 유지 + 투자 조언으로 비칠 법적 리스크 회피. 시간이 남으면 추가 검토

## Open Questions
- 4계절(봄/여름/가을/겨울)의 정확한 판정 기준 매핑과 각 계절의 톤/카피는 `/write-spec`에서 확정
- 결과 페이지에 근거를 얼마나 상세히 보여줄지(지표별 한 줄 vs 더 상세)는 `/write-spec`에서 확정
- Gemini API 에러/판정 실패 시 사용자에게 보여줄 화면은 `/write-spec`에서 확정
